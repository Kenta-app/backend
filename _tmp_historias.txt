ÉPICA 1 — Ingesta de datos 
HU-01 Ingesta desde fuentes externas Como administrador del sistema, 
quiero que el sistema ingeste noticias automáticamente desde 
sitios web de noticias peruanas y cuentas de Twitter/X 
configuradas, 
para disponer de un flujo continuo de contenido político 
actualizado sin intervención manual. 
Dado que existe al menos una fuente activa configurada en la BD, 
cuando se ejecuta el job de ingesta (manual o programado), 
entonces el sistema invoca la estrategia correcta (WebScraperIngestion o 
TwitterApiIngestion) según el tipo de la fuente, persiste los artículos en raw.news_raw 
con status='pending' y registra un IngestionLog con status='success' o 'failed'. 
 
Dado que la fuente es inaccesible (timeout, 4xx/5xx), 
cuando el scraper intenta conectarse, 
entonces el log queda con status='failed', se registra el mensaje de error y no se persiste 
ningún RawNews parcial. 
 
Dado que se completa una ingesta exitosa, 
cuando finaliza el proceso, 
entonces cada RawNews contiene: sourceId, platform, originalUrl, titleRaw, contentRaw, 
publishedAt y scrapedAt.

HU-02 Detección de duplicados Como administrador del sistema, 
quiero que el sistema detecte y descarte artículos duplicados 
antes de procesarlos, 
para evitar que una misma noticia aparezca múltiples veces 
en el pipeline y contamine los clusters. 
Dado que se ingesta un artículo cuya combinación (sourceId, originalUrl) ya existe en 
raw.news_raw, 
cuando el IngestionService ejecuta la deduplicación, 
entonces el artículo no se persiste y el log contiene el conteo de duplicados descartados. 
 
Dado que se ingesta un artículo nuevo (URL no vista), 
cuando se verifica la unicidad, 
entonces se persiste normalmente con status='pending'. 
 
Dado que el sistema procesa un batch de 100 artículos con 30 duplicados, 
cuando termina la ingesta, 
entonces solo 70 quedan en raw.news_raw y el IngestionLog refleja los 30 descartados.

ÉPICA 2 — Procesamiento NLP 
HU-04 Limpieza y normalización de 
texto 
Como administrador del sistema, 
quiero que el pipeline limpie y normalice el texto crudo de 
cada noticia ingestada, 
para obtener un texto de calidad que los modelos NLP 
puedan procesar con alta precisión. 
Dado que existe un RawNews con status='pending', 
cuando el PreprocessingService lo procesa, 
entonces se crea un ProcessedNews con cleanText sin HTML, URLs, caracteres 
especiales ni espacios redundantes, con language detectado y tokenCount calculado. 
 
Dado que el cleanText resultante tiene menos de 50 tokens, 
cuando se evalúa la procesabilidad, 
entonces el ProcessedNews queda con status='rejected' y se registra el motivo en 
ProcessingLog. 
 
Dado que se procesa un artículo exitosamente, 
cuando termina el PreprocessingService, 
entonces el RawNews original pasa a status='processed' y existe exactamente un 
ProcessingLog con stage='preprocessing' y status='success'.

ÉPICA 3 — Modelado ML 
HU-05 Preparación del dataset de 
entrenamiento 
Como científico de datos, 
quiero construir y versionar un dataset etiquetado con 
sentimiento (positivo/negativo/neutro) y veracidad (real/fake) 
de noticias políticas peruanas, 
para entrenar el modelo multitask con datos representativos 
del dominio objetivo. 
Dado que se recopilan textos de noticias políticas peruanas, 
cuando se aplica el proceso de etiquetado, 
entonces el dataset contiene mínimo 1 000 registros, con distribución documentada de 
clases por cada tarea y sin textos duplicados. 
 
Dado que el dataset está construido, 
cuando se divide en splits, 
entonces existe train (70%), validation (15%) y test (15%) estratificados por clase. 
 
Dado que el dataset es versionado, 
cuando se consulta su metadata, 
entonces se puede reproducir exactamente el mismo split con una semilla fija 
documentada. 
8 Épica 3: Modelado ML 
HU-06 Entrenamiento del modelo 
multitask 
Como científico de datos, 
quiero entrenar un modelo NLP multitask que prediga 
simultáneamente sentimiento y fake score sobre noticias 
políticas, 
para minimizar latencia de inferencia y aprovechar el 
aprendizaje compartido entre tareas. 
Dado que el dataset de entrenamiento está disponible, 
cuando se ejecuta el script de entrenamiento, 
entonces el modelo converge (pérdida de validación deja de decrecer más del 0.5% en 3 
epochs consecutivos) y se guarda con su modelVersion versionado. 
 
Dado que el entrenamiento finaliza, 
cuando se evalúa sobre el split de validación, 
entonces el F1-macro de sentimiento ≥ 0.70 y el AUC-ROC de fake score ≥ 0.75. 
 
Dado que existe un modelo previo, 
cuando se entrena una nueva versión, 
entonces ambas versiones quedan guardadas y el sistema permite seleccionar cuál usar 
en inferencia.

HU-08 Evaluación del modelo entrenado Como científico de datos, 
quiero evaluar el modelo sobre el split de test con métricas 
estándar y compararlo contra un baseline, 
para validar que cumple los umbrales mínimos de calidad

cuando se ejecuta la evaluación,

ROC, precisión en umbral 0.5 para fake score. 
 
Dado que las métricas son calculadas,

entonces el modelo multitask supera al baseline en todas las métricas reportadas.

junto al modelVersion evaluado.

ÉPICA 4 — Resumen 
HU-09 Generación automática de 
resúmenes 
Como administrador del sistema, 
quiero que el pipeline genere automáticamente un resumen 
breve de cada noticia representativa de un cluster, 
para ofrecer a los usuarios jóvenes un texto conciso que les 
permita comprender la noticia sin leer el artículo completo. 
Dado que existe un ProcessedNews representativo sin Summary, 
cuando el SummarizationService ejecuta generateSummary(), 
entonces se crea un Summary con summaryText de entre 50 y 150 palabras, 
modelVersion y createdAt. 
 
Dado que el summaryText generado está vacío o tiene menos de 20 palabras, 
cuando el sistema lo valida con isEmpty(), 
entonces la noticia no pasa a PublishedNews hasta que se regenere el resumen. 
 
Dado que el administrador solicita regenerar un resumen con otro modelo, 
cuando se invoca regenerateSummary() con la nueva modelVersion, 
entonces el summaryText se actualiza y la versión anterior queda registrada en 
ProcessingLog.

ÉPICA 5 — Backend / API 
HU-10 Exposición de resultados ML vía 
API REST 
Como desarrollador frontend, 
quiero consumir un endpoint REST que retorne noticias 
publicadas con sus predicciones ML y resumen, 
para construir el feed y el detalle de noticia sin lógica de 
negocio en el cliente. 
Dado que existe al menos una PublishedNews, 
cuando se hace GET /news con parámetros page y pageSize, 
entonces la respuesta retorna HTTP 200 con lista paginada que incluye: newsId, title,

Dado que se hace GET /news/{newsId} con un ID existente, 
cuando el NewsController procesa la petición, 
entonces retorna HTTP 200 con todos los campos de PublishedNews. 
 
Dado que se solicita un newsId inexistente, 
cuando el sistema busca el registro, 
entonces retorna HTTP 404 con mensaje estandarizado por errorResponse().

HU-11 Ejecución manual del pipeline Como administrador del sistema, 
quiero disparar manualmente el pipeline completo (ingesta → 
preprocesamiento → clustering → resumen → predicción → 
publicación) para una fuente específica, 
para reprocesar noticias o actualizar el feed sin esperar el job 
programado. 
 
 
Dado que el administrador autenticado envía POST /pipeline/run con un sourceId válido, 
cuando el PipelineController orquesta los servicios, 
entonces cada etapa se ejecuta en orden, cada una deja su ProcessingLog, y la 
respuesta final incluye el conteo de noticias publicadas. 
 
Dado que una etapa del pipeline falla,

log indica la etapa fallida con su mensaje.

cuando BaseController ejecuta requireRole('admin'), 
entonces retorna HTTP 403.

ÉPICA 10 — Clusters 
HU-12 Clustering de noticias similares Como administrador del sistema, 
quiero que el pipeline agrupe automáticamente noticias 
similares en clusters basados en similitud semántica, 
para eliminar redundancia en el feed y mostrar al usuario una 
sola noticia representativa por tema. 
Dado que existen ProcessedNews con status='processed' para una fuente, 
cuando el ClusteringService ejecuta clusterProcessedNews(), 
entonces cada ProcessedNews queda asignado a exactamente un ClusterMember, y 
cada NewsCluster tiene al menos un miembro. 
 
Dado que se forman los clusters, 
cuando se calcula clusterScore, 
entonces el score refleja la cohesión interna del cluster (similitud promedio entre 
miembros ≥ umbral configurable). 
 
Dado que el clustering falla por insuficiencia de datos (menos de 2 artículos), 
cuando el ClusteringService evalúa la condición, 
entonces no se crean clusters y se registra el motivo en ProcessingLog. 
8 Épica 10: Clusters 
HU-13 Selección de noticia 
representativa 
Como administrador del sistema, 
quiero que el sistema seleccione automáticamente la noticia 
más representativa de cada cluster, 
para que el resumen y las predicciones ML se apliquen sobre 
el artículo de mayor calidad y cobertura del tema. 
Dado que un NewsCluster tiene al menos un ClusterMember, 
cuando el ClusteringService ejecuta assignRepresentative(), 
entonces el cluster tiene exactamente un representativeNewsProcessedId apuntando al 
ProcessedNews con mayor similitud promedio con el resto del cluster. 
 
Dado que el representativo es seleccionado, 
cuando se inician las etapas de resumen y predicción, 
entonces solo ese ProcessedNews es enviado a SummarizationService y 
PredictionService. 
 
Dado que el representativo actual es eliminado del sistema, 
cuando el ClusteringService detecta la situación, 
entonces reasigna el representativo al siguiente candidato según el criterio de similitud.

ÉPICA 7 — Personalización 
HU-19 Recomendación basada en 
historial (opcional – en el futuro) 
Como usuario autenticado, 
quiero recibir noticias recomendadas en base a mis 
interacciones previas (vistas, clics y reacciones), 
para descubrir contenido político relevante para mis intereses 
sin tener que buscarlo manualmente. 
Dado que el usuario tiene al menos 5 interacciones registradas (NewsView o NewsClick), 
cuando accede a la sección 'Para ti', 
entonces el sistema retorna hasta 10 noticias ordenadas por score de relevancia 
personal. 
 
Dado que el usuario no tiene historial suficiente (< 5 interacciones), 
cuando accede a la sección 'Para ti', 
entonces se muestran las noticias más recientes como fallback. 
 
Dado que el usuario tiene historial, 
cuando el sistema calcula recomendaciones, 
entonces las noticias ya vistas con isEngagedView(10) no se vuelven a recomendar.

ÉPICA 8 — Analítica 
HU-20 Métricas de engagement de 
usuarios 
Como administrador del sistema, 
quiero consultar métricas agregadas de interacción (vistas 
totales, clics, tiempo promedio, ratio de reacciones 
positivas/negativas) por noticia y por período en un 
dashboard, 
para evaluar el impacto de la plataforma y tomar decisiones 
sobre el contenido. 
Dado que existen registros de NewsView, NewsClick y NewsReaction, 
cuando el administrador consulta GET /admin/analytics/engagement con un rango de 
fechas, 
entonces retorna: total de vistas, clics, tiempo promedio de lectura, y ratio 
positivo/negativo por noticia en el período. 
 
Dado que se filtra por newsId específico, 
cuando el sistema procesa la consulta, 
entonces retorna las métricas desagregadas solo para esa noticia. 
 
Dado que no hay interacciones en el período consultado, 
cuando el sistema responde, 
entonces retorna los campos en cero (no un error 404). 
5 Épica 8: Analítica 
HU-21 Métricas de rendimiento del 
modelo ML 
Como administrador del sistema, 
quiero consultar el historial de métricas de evaluación por 
versión del modelo ML en un dashboard, 
para comparar versiones y decidir cuál se mantiene activa en 
producción. 
Dado que existen múltiples MlPredictions con distintos modelVersion, 
cuando el administrador consulta GET /admin/analytics/model, 
entonces retorna por cada versión: conteo de predicciones, distribución de

Dado que se solicita comparación entre dos versiones de modelo, 
cuando el sistema procesa la consulta,

Dado que solo existe una versión del modelo, 
cuando se solicita comparación, 
entonces retorna HTTP 400 indicando que se requieren mínimo dos versiones.
